#+title: org-dex-parse
#+author: Gabriele Della Vecchia
#+startup: overview

Org-mode parsing library for [[https://github.com/gdvek/org-dex][org-dex]].  Transforms org files into structured
items using [[https://github.com/karlicoss/orgparse][orgparse]] as the parsing backend.

Successor of [[https://github.com/gdvek/remi-org-parse][remi-org-parse]] — same mission, two key evolutions:
configurable item definition (not hardcoded) and fixes for all parser bugs
discovered during [[https://github.com/gdvek/remi-org-db][remi-org-db v3]] regression testing.

* What it does
  Given an org file and a configuration, produces a flat tuple of parsed items.
  Each item is a heading that matches a user-defined predicate — not a hardcoded
  rule.

  #+begin_example
  org file ──→ org-dex-parse ──→ ParseResult
                    │                 └── .items: tuple[Item, ...]
                    │
               Config
                 .item_predicate   ← what counts as an item
                 .todos / .dones
                 .extra_tag_chars  ← custom tag characters
                 .exclude_drawers
                 .exclude_blocks
                 ...
  #+end_example

* How it differs from remi-org-parse

  | Aspect              | remi-org-parse                  | org-dex-parse              |
  |---------------------+---------------------------------+----------------------------|
  | Item definition     | Hardcoded: =:ID:= + =:Type:=        | Configurable predicate     |
  | Consumer            | remi-org-db (Python)            | org-dex (Elisp)            | /[Org dex is in python]/
  | Drawer exclusion    | One policy (body + links)       | Separate policies          |
  | Timestamp scoping   | Broad (drawer/planning leakage) | Zone-aware extraction      |
  | Link extraction     | Loses links in excluded drawers | Separate body/link policy  |
  | Link classification | 6 hardcoded categories          | None — consumer classifies |
  | Tag characters      | Hardcoded monkey-patch          | Configurable extra chars   |
  | Body markup         | Stripped to plain text          | Raw text preserved         |
  | Property scoping    | Subtree traversal               | Direct drawer only         |
  | Performance         | Multi-pass tree walk            | Optimized (fewer passes)   |

  Full list of v2 bugs addressed: [[file:project/v2-parser-findings.org][v2-parser-findings.org]].

* Architecture
  =orgparse= is a dependency, not a fork.  It handles low-level org-mode syntax
  (headlines, properties, planning, timestamps).  This package adds the domain
  layer: item discrimination via predicate, zone-aware extraction, body
  collection with proper scoping.

** Modules
   
   | Module    | Responsibility                                       |
   |-----------+------------------------------------------------------|
   | =parser.py= | Orchestration: load file → walk tree → extract items |
   | =body.py=   | Body text collection, affiliated keyword exclusion   |
   | =links.py=  | Link extraction (zone-aware), no classification      |
   | =config.py= | =Config=: predicate, keywords, exclusion lists         |
   | =types.py=  | =Item=, =Timestamp=, =Link=, =ParseResult=                   |

** Item predicate
   The predicate is a function =headline → bool= that determines which
   headings are items.  =:ID:= is a structural invariant — the predicate is
   only called on headings that already have =:ID:=.

   #+begin_example
   Predicate examples:

   canonical:  lambda h: True              (any heading with :ID:)
   remi:       lambda h: has_property(h, "Type")  (:ID: + :Type:)
   roam-like:  lambda h: not has_property(h, "ROAM_EXCLUDE")
   #+end_example

   The predicate partitions the entire org tree.  Every heading belongs to
   exactly one item: either it /is/ an item, or it belongs to its nearest
   item ancestor.  This partitioning determines:
   - parent-item-id (transparent scaffolding traversal)
   - body collection boundaries
   - timestamp/link extraction scope

** Zone-aware extraction
   Extraction of timestamps, links, and body text respects exclusion zones.
   A zone is a structural region (drawer, block, planning line) where certain
   data types should not be extracted from.

   #+begin_example
   ┌─ heading ──────────────────────────────────┐
   │  ┌─ PROPERTIES ──────┐  always excluded    │
   │  │  :ID: ...         │  from ts/link/body  │
   │  └───────────────────┘                     │
   │  ┌─ LOGBOOK ─────────┐  excluded from body │
   │  │  clock, states    │  extracted by       │
   │  │                   │  dedicated handlers │
   │  └───────────────────┘                     │
   │  ┌─ ___LINKED_BY___ ┐  excluded from body  │
   │  │  [[id:...][...]]  │  links EXTRACTED    │
   │  └───────────────────┘                     │
   │  body text here ...     ← body, ts, links  │
   └────────────────────────────────────────────┘
   #+end_example

   Key principle: "exclude from body" ≠ "exclude from links".  Exclusion
   policies are per data-type, not global.

** Extract, don't classify
   The parser extracts structured data but does not classify it into
   domain-specific categories.  Classification is the consumer's job.

   #+begin_example
   ┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
   │   org file        │     │  org-dex-parse   │     │  consumer        │
   │                   │ ──→ │  (extract)       │ ──→ │  (classify)      │
   │  :tag1: :tag2:   │     │                   │     │                   │
   │  [[id:...][...]] │     │  .local_tags      │     │  tag_category()  │
   │  [[https://...]] │     │  .links           │     │  link_category() │
   └──────────────────┘     └──────────────────┘     └──────────────────┘
   #+end_example

   *Tags:* extracted as raw strings in =.local_tags= and =.inherited_tags=.
   The consumer applies its own classification (by prefix, by mapping, or
   whatever logic fits the workflow).  This is the same pattern used in
   remi-org-db, where =tag_category()= classifies at indexing time.

   *Links:* extracted as =.links: tuple[Link, ...]=.  Each =Link= carries
   =.target= (the raw link target as it appears inside =[[...]]=, e.g.
   ="id:abc-123"=, ="https://example.com"=, ="Some heading"= for fuzzy)
   and =.description=.  The consumer extracts the schema and groups links
   into categories as needed.

   remi-org-parse classified links into 6 hardcoded categories (including
   =excalidraw_links=, specific to one workflow).  org-dex-parse removes
   this coupling: the parser is workflow-agnostic, the consumer owns the
   classification.

* Consumers
  - =org-dex= — indexing engine (primary consumer)
  - CLI tools — batch analysis, validation

* Limitations
  - *LOGBOOK drawer name is hardcoded.*  The timestamp filter excludes the
    =:LOGBOOK:...:END:= drawer (the default when =org-log-into-drawer= is =t=).
    Custom drawer names (=org-log-into-drawer= set to a string) and inline
    logging (=org-log-into-drawer= = =nil=) are not supported — logging
    timestamps would leak into =inactive_ts= as false positives.
